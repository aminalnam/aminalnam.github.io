OMEGA Architecture

Wireless Mesh Network

The transport layer binding the autonomous fleet together. OMEGA uses a robust Reticulum-inspired DTN overlay to push live telemetry back to the central gateway.

Reticulum-Inspired DTN Overlay

Marine environments are characterized by disrupted connectivity, high latency, and severe bandwidth constraints. Traditional protocols like MQTT—which expect a brokered client/server pattern over reliable TCP/IP streams—fail completely over a LoRa network spanning miles of ocean waves.

Instead, OMEGA separates the overlay network from the physical radio bearer. The software architecture utilizes a Delay-Tolerant Network (DTN) model with store-and-forward routing. This means a node doesn't need to reach the shoreline gateway directly; it can pack a telemetry bundle and pass it to a nearby floating relay. That relay will store the bundle and forward it later if the next hop is currently out of range.

Why Not MQTT Everywhere?

A common mistake in IoT sensor networks is trying to force MQTT over every link. In the OMEGA stack, MQTT is strictly reserved for the IP-capable edge (Gateways to Cloud).

The MQTT Trap

MQTT relies on TCP/IP. Pushing TCP handshakes over a 915MHz LoRa link with a 1% duty cycle is extremely inefficient and prone to timeouts. It forces tiny microcontrollers to maintain complex routing states and connection keepalives.

The Gateway Handoff

In OMEGA, sensor nodes send highly compact internal frames. The Shoreline Gateway receives these frames, validates them, and translates them into MQTT topics. The Gateway then publishes them (e.g., omega/fleet/rover-01/depth) over standard cellular/Wi-Fi to the web backend.

CBOR Payload Compression

To maximize the efficiency of the LoRa and acoustic links, JSON data must be aggressively compressed. OMEGA utilizes CBOR (Concise Binary Object Representation) to pack nested telemetry structures into dense byte streams.

Serialization Logic (Python Edge)

The following snippet demonstrates how the OMEGA gateway handles the translation between human-readable JSON payloads destined for the FastAPI portal and the raw CBOR streams sent over the LoRa physical bridge.

import cbor2
import json

def encode_telemetry_for_lora(json_payload: dict) -> bytes:
    """
    Compresses standard JSON telemetry into a dense CBOR structure
    suitable for transmission over the LilyGO T-Beam LoRa link.
    """
    try:
        # Example payload: {"rover_id": 4, "lat": 34.05, "lon": -118.24, "depth": 14.2}
        cbor_bytes = cbor2.dumps(json_payload)
        
        # Verify payload size is under LoRa MTU (typically 250 bytes)
        if len(cbor_bytes) > 250:
            raise ValueError("Payload exceeds LoRa maximum transmission unit")
            
        return cbor_bytes
        
    except Exception as e:
        log_error(f"CBOR Compression failed: {e}")
        return b""

def decode_lora_to_api(cbor_bytes: bytes) -> dict:
    """
    Restores the binary CBOR payload received at the Shoreline Gateway
    back into a JSON structure for FastAPI ingestion.
    """
    decoded_dict = cbor2.loads(cbor_bytes)
    return json.dumps(decoded_dict)

Self-Healing Mesh Topologies

By treating the underlying radio hardware purely as a dumb bearer layer, the software mesh can route around failures dynamically. If a HaLow IP link drops due to distance, the software stack seamlessly downshifts to transmitting smaller CBOR bundles over the LoRa network, ensuring critical health and positional telemetry always makes it back to the portal.
