Control Systems Engineering

Sensor Array Fusion.

Deploying autonomous vessels physically into the ocean requires a rigid, unbreakable separation of concerns. If a high-level Python script crashes, the edge node cannot be allowed to corrupt the local NVRAM or drain the battery.

The OMEGA Fleet node profile isolates safety-critical power management from high-level, unpredictable mission routing.

Relevance: A dedicated hardware watchdog timer acts as the fail-safe arbiter. If the primary compute layer crashes or hangs due to a memory leak, the watchdog timer mathematically cuts the power rail and forces a hard reboot, guaranteeing the asset recovers without human intervention.

Mission Intents vs Raw AT Commands

Operators do not send raw AT modem commands over the mesh. They send abstract "Mission Intents" (e.g., "Increase Sampling Rate"), which the primary compute layer expands into local hardware configurations.

gateway/node_controller.py

def receive_intent(intent: MissionIntent):
    if intent.type == "survey_grid":
        waypoints = generate_coverage_path(intent.polygon)
        hardware_watchdog.update_profile(waypoints)
        return {"status": "executing"}
Hardware Physics

Thermodynamics & Heat Dissipation.

Deploying powerful microcontrollers into marine environments introduces severe, often fatal hardware constraints. Edge nodes must be encased in completely sealed, hermetic IP67+ waterproof enclosures to survive saltwater immersion. However, processors like the ESP32 and accompanying 30dBm LoRa amplifiers generate significant internal heat.

Because they are sealed, there is zero active ventilation. The system relies entirely on passive thermal conduction through the enclosure walls to the surrounding ambient air or water. If the firmware transmits data continuously, the internal ambient temperature will inevitably rise until the silicon hits its thermal throttling limit, leading to catastrophic system failure or melted PLA mounts.

How we use this: OMEGA firmware implements rigid software limiters on radio transmit duty-cycles. It monitors the internal CPU die temperatures and aggressively throttles clock speeds or drops non-critical telemetry packets to ensure the thermal envelope of the enclosure is never breached.

Marine Physics

Archimedes' Principle & Wind Drag.

A common question is: "Why don't you just put a massive 40-lb car battery in the buoy to run the radio forever?" The answer is brutal physical constraint.

Archimedes' Law

Buoyancy Deficit

To float a 40-lb lead-acid battery, Archimedes' Principle mathematically requires displacing 40 lbs of saltwater. This demands an incredibly massive, bulky foam hull.

Aerodynamics

Wind Shear

A massive foam hull exponentially increases the surface area exposed to wind drag. In a 15-knot ocean breeze, the buoy acts like a giant sail.

System Failure

Station-Keeping Loss

The massive wind drag would overcome the mooring line, dragging the anchor across the reef. We MUST use lightweight 18650 lithium cells to keep the hydrodynamic profile razor-thin.

Deployment Operations

Over-The-Air (OTA) Pipelines.

Retrieving a deployed buoy or a remote sensor from the middle of the ocean simply to update a line of code is logistically and financially prohibitive. OMEGA implements a robust Over-The-Air (OTA) flashing pipeline over its own DTN mesh network.

How it works: The compiled C++ firmware binaries are fragmented, heavily compressed, and signed with a cryptographic private key. The Gateway transmits these fragments sequentially over the LoRa network. The edge node receives the fragments, buffers them in its external SPI flash memory, and reassembles them.

Crucially, the node then mathematically verifies the cryptographic signature of the reassembled binary. It only switches the boot partition pointer to the new firmware if the validation passes perfectly, completely preventing "bricked" nodes from corrupt RF packets.

Fleet Algorithms

Autonomous Relay Deployment Logic.

The edge node does not rely on a human operator to decide when to expand the network. It autonomously executes Relay Deployment Logic. The build_relay_chain_plan Python function continuously monitors the RF environment.

The Math: It calculates the Haversine distance between the shore and the edge node. However, RF propagation over rolling waves is not a simple linear equation. The algorithm actively monitors the direct_rssi_dbm (Received Signal Strength Indicator) of the LoRa packets. If the signal decays below the strict deploy_rssi_dbm threshold (e.g., -115 dBm), the node triggers a physical servo to drop a floating LoRa relay buoy. This proves the edge node is reacting to live, unpredictable RF physics rather than simple geographic distance assumptions.

OTA Updates

Live Firmware Flasher.

Standard IoT platforms require operators to download bin files and physically plug USB cables into hardware to update firmware. OMEGA completely automates this via the Live Firmware Flasher directly within the Mission Portal.

How it works: The Python backend uses the subprocess module to dynamically execute platformio CLI commands. This allows operators to select targeted hardware profiles (like [env:tbeam-supreme-usb-telemetry]), compile the C++ code on the fly inside the gateway, and push the resulting binaries either over the air (OTA) via the mesh network or directly via a tethered USB port, without ever leaving the web browser.

firmware_flasher.py
def build_firmware_command(target: FirmwareTarget, *, action: FirmwareAction, port: str | None = None) -> list[str]:
    command = [
        *_platformio_command(),
        "run", "-d", str(target.project_dir), "-e", target.environment,
    ]
    if action == "upload":
        command += ["-t", "upload", "--upload-port", port]
    return command
Autonomous Safety

Active Guard Geofencing.

The active_guard.py module acts as a merciless, unblinking watchdog over the physical fleet. During high winds or heavy chop, a remote sensor might struggle to maintain its commanded heading and slowly drift off-course.

Execution: Active Guard constantly calculates the cross-track error (the perpendicular distance from the planned path) of every edge node against predefined GeoJSON exclusion zones. If a vessel mathematically deviates into a dangerous shipping lane or a shallow reef, Active Guard instantly overrides the standard waypoint mission, triggering an emergency acoustic beacon to alert recovery vessels before the node is lost.

Simulation

Digital Twin Execution.

Deploying a massive mesh network in the ocean without testing is guaranteed to fail. Before a fleet is ever put on a boat, operators launch a shadow fleet using the digital_twin.py architecture.

How it works: This module spawns hundreds of virtual edge nodes running the exact same C++ firmware logic. These digital twins mimic the physical hardware quirks, specific battery degradation discharge curves, and exact LoRa RF transmission patterns of the physical vessels. This completely stress-tests the Mesh Core, the FastAPI routing, and the SQLite WAL configuration before the hull ever touches the saltwater.

Hardware Architecture

Raspberry Pi Coprocessor.

An ESP32 microcontroller is incredibly efficient at radio transmissions, but it physically lacks the RAM and clock speed to run a complex asynchronous routing table or a SQLite database. The OMEGA architecture utilizes a hardware tether.

Execution: The pi_gateway_controller.py and gateway_coprocessor_bridge.py modules dictate this relationship. A Raspberry Pi 4 acts as the "brain," handling all HTTP requests, database transactions, and Python-based routing logic. The tethered ESP32 (connected via USB serial) acts merely as a "dumb" RF modem. The Pi injects specific AT commands and byte arrays into the ESP32, which simply blasts them out over the LoRa antenna.

Deterministic Builds

PlatformIO Matrices.

The Arduino IDE is wholly inadequate for managing complex, multi-layered C++ mesh networks across a dispersed engineering team. OMEGA firmware fundamentally relies on PlatformIO (platformio.ini) to enforce deterministic, repeatable builds.

Execution: The project defines strict build environments for entirely different ESP32 hardware profiles (e.g., LilyGO T-Beam vs Heltec V3). Specific compiler flags like -DOMEGA_RADIO_KIND_LR1121=1 dynamically instruct the C++ preprocessor. This ensures that when the compiler runs, it seamlessly configures the correct internal SPI pinouts and power profiles for a Semtech LR1121 transceiver versus a standard SX1262, preventing hardware mismatch crashes.

platformio.ini
[env:gateway_t3_s3_lr1121_subghz]
platform = espressif32
board = esp32-s3-devkitc-1
framework = arduino
build_flags =

  -DOMEGA_ENABLE_RADIOLIB=1
  -DOMEGA_RADIO_KIND_LR1121=1

  -DOMEGA_LORA_TX_DBM=17
  -DRADIO_NSS=7
  -DRADIO_DIO1=36
