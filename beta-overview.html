Testing Framework

BETA Framework.

The Build Environment for Testing & Analytics (BETA). A standalone deterministic evaluation engine that continuously subjects backend APIs and gateways to high-frequency synthetic telemetry, measuring pipeline latency, detecting performance regressions, and generating AI-driven remediation tickets.

View OMEGA Integration View Test Suite
The Methodology

Evidence-Based Engineering.

When building a complex system that sits between unpredictable physical edge hardware and high-frequency backend databases, "it works on my machine" is an unacceptable standard. Every commit pushed to the target gateway must be empirically proven to handle chaotic telemetry floods without locking the SQLite WAL or dropping HTTP connections.

Telemetry Generator

Deterministic Fuzzing

BETA generates thousands of synthetic payload injections—simulating multiple drifting devices, acoustic anomalies, and deliberate malformed hardware packets.

Performance Matrices

Tracking Regressions

The system tracks the P95 Query Latency and Ingest Throughput over time. If a new software build drops the ingestion rate below 250 obs/sec, a regression flag is thrown.

Automated Remediation

AI Copilot Ticketing

Post-run, the Honu AI Agent analyzes the performance deltas. If a regression is detected, it automatically writes a Jira-style technical ticket outlining the suspected bottleneck.

Deterministic Fuzzing & Race Conditions

A gateway might process a single incoming buoy payload perfectly. But what happens when an underwater acoustic anomaly triggers 50 edge nodes to transmit data at the exact same millisecond? Without rigorous testing, "race conditions" can lock up the database and drop critical scientific data.

The Core Concept

Deterministic Fuzzing

To guarantee system stability before physical deployment, we must subject the backend to extreme stress. "Fuzzing" involves automatically injecting thousands of synthetic, highly-concurrent, and occasionally malformed telemetry packets to see if the server crashes. Because the payloads are deterministically generated, any bugs discovered are completely reproducible.

The Security Flaw

SQLite WAL Locking

When 50 parallel requests try to write to a SQLite database simultaneously, you will hit a database is locked exception. This deep dive demonstrates how to use Python's asyncio to simulate this exact failure mode, proving the necessity of Write-Ahead Logging (WAL) and asynchronous background tasks.

Step-by-Step Implementation: The Asynchronous Injector
  1. The Generator: The `CborPayloadGenerator` creates mathematically valid but completely randomized buoy sensor data.
  2. The Flood Loop: We use an asynchronous `while` loop bound by a specific duration to continuously generate payloads.
  3. Fire and Forget: By using `asyncio.create_task()`, we launch the HTTP POST request into the background without waiting for a response, allowing us to hit the gateway with hundreds of concurrent connections per second.
import asyncio
from core.fuzzer import CborPayloadGenerator
from network.client import AsyncGatewayClient

async def run_synthetic_flood(target_url: str, duration_sec: int, tps: int):
    """Fires a sustained flood of synthetic marine telemetry."""
    client = AsyncGatewayClient(target_url)
    generator = CborPayloadGenerator(node_count=50)
    
    print(f"🌊 Initiating flood: {tps} TPS for {duration_sec}s")
    
    end_time = asyncio.get_event_loop().time() + duration_sec
    while asyncio.get_event_loop().time() < end_time:
        payload = generator.create_drift_telemetry()
        
        # Fire-and-forget payload injection (simulating parallel buoys)
        asyncio.create_task(client.post_telemetry(payload))
        await asyncio.sleep(1.0 / tps)
        
    await client.close()
    return client.get_latency_metrics()

Performance Metrics (P95) & Remediation

When a synthetic flood finishes, how do we grade the system? Tracking the "average" response time is mathematically flawed because outliers are hidden. Instead, Site Reliability Engineering (SRE) relies on percentiles.

The Core Concept

P95 Latency

P95 latency means "95% of all requests completed faster than this time." If your average latency is 10ms, but your P95 is 500ms, it means 5% of your buoys are experiencing massive delays (likely due to a queue bottleneck). The BETA framework automatically fails any code commit that degrades the P95 ingestion metric beyond acceptable limits.

Implementation Guide

AI-Driven Triage

When the P95 latency spikes, debugging the backend is tedious. The BETA framework feeds the performance deltas, CPU profiles, and the recent Git commit diff directly into the Honu AI Copilot. The AI analyzes the bottleneck and generates a ready-to-work Jira ticket for the engineering team.

⚠️ Example Output: AI Auto-Triage Ticket
Title: [REGRESSION] P95 Latency spike to 120ms during concurrent SQLite writes
Hypothesis:
The recent commit (fc92b4) introduced an asynchronous index update on the `telemetry_data` table. Because the database is in WAL mode, the prolonged index rebuild is starving the synchronous writer pool, leading to connection timeouts from the Synthetic Injector.

Recommended Remediation:
1. Move the index update to a background worker queue (e.g., Celery) rather than blocking the main FastAPI thread.
2. Ensure `PRAGMA synchronous = NORMAL` is strictly enforced during high-throughput ingestion windows.
