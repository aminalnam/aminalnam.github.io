How OMEGA Works

OMEGA can be used to teach a connected set of ideas in marine science, physics, measurement, and mapping. The sections below explain the main concepts in the system, show where they appear in the code, and clarify how those ideas shape the final bathymetric map.

Acoustic Depth Data Validity Temperature and Sound Speed Depth and Position Interpolation
Concept 1

Acoustic depth measurement

OMEGA measures depth by using sound to estimate the distance between the sensor and the bottom.

The idea

The sonar sends an acoustic pulse downward through the water. That pulse reflects off the bottom and returns to the sensor. The system uses the travel time of that pulse, together with an assumed sound speed, to estimate distance.

depth = (sound speed × travel time) / 2

The division by two matters because the pulse travels the path twice: once downward and once back upward. The sensor is measuring total round-trip time, but the goal is one-way distance to the bottom.

In OMEGA

In the logger, the decoded sonar reading is first turned into centimeters, then corrected, checked, and added to the depth stream. That means the number coming from the sensor is only one stage in the process. The system still has to decide how that reading should be interpreted and whether it fits the accepted operating range.

Code example

// Convert the decoded sonar packet into centimeters.
float reportedDepthCm = raw / 10.0f;

// Apply the temperature-based sound-speed correction.
float correctedDepthCm = correctSonarRangeCm(reportedDepthCm, tempC);

// Keep only values that fall inside the accepted operating range.
if (correctedDepthCm >= 2.0f && correctedDepthCm <= 800.0f) {
  // Add the corrected depth to the smoothing buffer.
  addDepthSample(correctedDepthCm);
}

This snippet shows that depth is not simply read and stored. The value is converted, corrected, constrained to a usable range, and then added to the working dataset.

Why this matters

This concept is a strong entry point into underwater sensing because it shows how a physical principle becomes a scientific tool. OMEGA is not touching the bottom directly. It is inferring bottom distance from the behavior of a wave traveling through water.

It also introduces the idea that an instrument reading is often the result of a model. The depth value depends on assumptions about the medium and on how the signal is processed after it is received.

Concept 2

Why data validity matters

OMEGA does not treat every reading as equally reliable. It checks whether a point is strong enough to become part of the map.

The idea

A sensor can return a number even when survey conditions are poor. In bathymetry, a useful point needs more than a depth reading. The position has to be trustworthy, the depth has to be stable, and the platform has to be operating within acceptable motion and orientation limits.

This is why validity matters so much in a mapping system. A weak point does not stay isolated. Once it enters the mapping dataset, it can influence interpolation, contour shape, and the appearance of the final surface.

In OMEGA

OMEGA makes these checks explicit. The code examines GPS readiness, number of satellites, horizontal dilution of precision, fix age, depth readiness, platform speed, pitch, roll, and IMU state. Only points that pass those checks are considered ready for mapping.

Code example

const char* getMapStatus(unsigned long currentMillis) {
  if (millis() - startupMillis < gpsWarmupTime) return "WARMUP";
  if (!mapReady || !sdReady || !mapFile) return "NOFILE";
  if (!gpsDataPresent(currentMillis)) return "NOGPS";
  if (!gps.location.isValid()) return "NOFIX";
  if (!gps.date.isValid() || !gps.time.isValid()) return "NOTIME";
  if (isnan(depthCm)) return "NODEP";
  if (depthSamplesCollected < 3) return "WARM";
  if (!gps.satellites.isValid()) return "SAT";
  if (gps.satellites.value() < minMapSats) return "SAT";
  if (!gps.hdop.isValid()) return "HDOP";
  if (gps.hdop.hdop() > maxMapHdop) return "HDOP";
  if (gps.location.age() > maxMapFixAgeMs) return "STALE";
  if (!isnan(smoothedSpeed) && smoothedSpeed > maxMapSpeedKmph) return "FAST";

  if (requireLevelForMapping) {
    if (!imuReady) return "NOIMU";
    if (!imuOrientationValid) return "IMU?";
    if (imuAccuracy < minImuAccuracyForMapping) return "CAL";
    if (fabs(imuPitchDeg) > maxMapPitchDeg || fabs(imuRollDeg) > maxMapRollDeg) return "UNLVL";
  }

  return "ARMED";
}

The value returned by this function is a compact summary of point quality. It lets the system distinguish between a point that is merely present and a point that is actually usable in the survey.

Why this matters

This is one of the clearest concepts for teaching scientific measurement. The issue is not only whether the instrument can produce numbers. The issue is whether those numbers are good enough to support interpretation.

In field science, quality control is part of the measurement process itself. OMEGA makes that visible in code, which is useful because students can see that reliability is not abstract. It is expressed through thresholds, checks, and decisions about what counts as usable evidence.

Concept 3

Temperature changes the depth reading by changing sound speed

OMEGA records temperature because the speed of sound in water is not fixed. That affects how sonar depth should be interpreted.

The idea

Sonar depends on travel time, but travel time only becomes distance when the system knows what sound speed to assume. If sound is moving through colder or warmer water than the calibration condition, the resulting depth estimate shifts.

Temperature matters because it changes the physical behavior of the medium through which the signal travels. The water is not just a setting around the measurement. It is part of the measurement model.

In OMEGA

OMEGA uses the measured water temperature to adjust the baseline sound speed used during calibration. The corrected sound-speed estimate is then used to rescale the reported sonar range.

Code example

float estimateSoundSpeed(float waterTempC) {
  float c = sonarCalSoundSpeed;

  if (!isnan(waterTempC)) {
    c += 4.6f * (waterTempC - sonarCalTempC);
  }

  return c;
}

float correctSonarRangeCm(float reportedRangeCm, float waterTempC) {
  float cActual = estimateSoundSpeed(waterTempC);
  return reportedRangeCm * (cActual / sonarCalSoundSpeed);
}
Calibration sound speed = 1500.0 Calibration temperature = 20°C
Measured temperature = 15°C Reported depth = 200 cm
Estimated sound speed = 1500 + 4.6 × (15 - 20) = 1477
Corrected depth = 200 × (1477 / 1500) ≈ 196.9 cm

In this example, cooler water lowers the estimated sound speed, which slightly lowers the corrected depth.

Why salinity is not included here

Salinity also affects sound speed, but OMEGA does not currently measure salinity directly. Temperature is already present in the system and provides a clear, teachable way to show how environmental conditions affect the calculation. In more advanced versions, salinity could be added to expand the correction model.

Why this matters

This concept helps explain that instruments often rely on partial models of the world. OMEGA is not trying to build the most complex sound-speed model possible. It is using a model that matches the available sensors and the goals of the system.

That makes this section useful for teaching both marine science and engineering. Students can see that real devices are built from tradeoffs: what variables are measured, what variables are omitted, and how those decisions affect the final result.

Concept 4

Depth becomes bathymetry when it is tied to position

A single depth reading can describe local water depth. Bathymetry begins when that depth is tied to coordinates and becomes part of a spatial dataset.

The idea

Bathymetry is not just the collection of depth values. It is the description of underwater shape across space. That means each depth measurement has to remain attached to where it was recorded.

This is why position matters so much in OMEGA. Without latitude and longitude, a depth value can tell you how deep the water was at one moment, but it cannot be placed onto a map, compared across a path, or used to reconstruct bottom form.

In OMEGA

The logger records time, coordinates, depth, temperature, and other conditions into a single survey record. Later, the mapping script separates latitude, longitude, and depth so those values can be placed into a grid and used for interpolation.

Why this matters

This section helps connect measurement to geospatial thinking. OMEGA is not only determining how deep the water is. It is building a set of measurements that describe where depth changes occur across an area.

It also clarifies why GPS quality appears so prominently in the validation logic. Position is not extra information around the depth reading. It is part of what makes the reading useful for bathymetry in the first place.

Concept 5

Interpolation builds a continuous surface from discrete points

OMEGA does not measure every location on the bottom. It measures selected points and uses interpolation to estimate the surface between them.

The idea

Survey data is made of individual points distributed across an area. A map, however, often needs to look continuous. Interpolation is the process that estimates values between measured locations so those discrete points can become a surface.

OMEGA uses inverse distance weighting, or IDW. In this approach, nearby points have more influence on an estimated location than distant points do.

estimated_value = sum(weight × known_value) / sum(weight) where weight = 1 / distance^power

In OMEGA

The Python mapping script first builds a grid over the surveyed area. It then calculates an estimated depth for each grid cell by weighting the surrounding measured points according to distance.

Code example

def idw_interpolation(x, y, z, xi, yi, power=2.0, epsilon=1e-12):
    zi = np.zeros_like(xi, dtype=float)

    for row in range(xi.shape[0]):
        for col in range(xi.shape[1]):
            dx = x - xi[row, col]
            dy = y - yi[row, col]
            dist = np.sqrt(dx * dx + dy * dy)

            if np.any(dist < epsilon):
                zi[row, col] = z[np.argmin(dist)]
                continue

            weights = 1.0 / np.power(dist, power)
            zi[row, col] = np.sum(weights * z) / np.sum(weights)

    return zi

def build_grid(points, grid_size, power):
    lats = np.array([p["lat"] for p in points], dtype=float)
    lons = np.array([p["lon"] for p in points], dtype=float)
    depths = np.array([p["depth_m"] for p in points], dtype=float)

    lon_min, lon_max = lons.min(), lons.max()
    lat_min, lat_max = lats.min(), lats.max()

    xi_vals = np.linspace(lon_min, lon_max, grid_size)
    yi_vals = np.linspace(lat_min, lat_max, grid_size)
    xi, yi = np.meshgrid(xi_vals, yi_vals)

    zi = idw_interpolation(lons, lats, depths, xi, yi, power=power)
    return lons, lats, depths, xi, yi, zi

The first function performs the weighted estimate. The second creates the grid that those estimates will fill.

Why this matters

This concept is important because it shows that the final map is a modeled surface, not a direct photograph of the bottom. The map depends on where points were collected, how many points exist, and what interpolation rule is used.

It also makes sampling visible. If points are sparse, uneven, or clustered, the surface can miss or smooth out real bottom features. That makes interpolation a useful way to teach both computational methods and the limits of what a map can claim to show.

OMEGA brings together acoustic sensing, environmental correction, measurement quality, geospatial positioning, and interpolation into one workflow that can be used to teach how underwater data becomes a map.
