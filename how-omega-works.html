OMEGA

How OMEGA works.

A guide to the structure of the system: what the Arduino logger measures, how points are accepted or rejected, and how Python turns those accepted points into bathymetric outputs.

Overview

OMEGA is organized as a measurement pipeline.

The Arduino Mega gathers sensor data in the field, checks whether a reading is usable for mapping, writes both raw and filtered logs to SD, and hands the accepted points to Python scripts that turn those points into map products.

Workflow

From field reading to map output

1. CaptureGPS, sonar, temperature, IMU, UTC time
2. ValidateFix quality, depth stability, motion, spacing
3. LogFull record plus accepted mapping dataset
4. ReconstructIDW surfaces, contours, KML, KMZ overlays
Why the split matters

OMEGA keeps the complete field record and a cleaner map dataset at the same time. That makes it possible to inspect what happened during the survey without forcing every recorded number into the final surface.

Codebase

Where the logic lives

  • arduino/rov_logger_mapping.ino handles sensor input, validation, display, and SD logging.
  • scripts/csv_to_png_depth_interpolated.py converts accepted points into an interpolated depth surface.
  • scripts/csv_to_png_depth_contours.py builds contour plots from the same mapping dataset.
  • scripts/csv_to_kmz_depth_overlay.py exports a georeferenced overlay for Google Earth.
Key ideas

The project brings several different concepts into one system.

Each part of OMEGA can be broken out and studied on its own, but the system becomes more useful when those parts are understood together: physics, environmental measurement, data quality, geospatial thinking, and interpolation.

Physics

Acoustic depth sensing

The sonar estimates distance from pulse travel time. That makes depth a time-based measurement that depends on the speed of sound in water.

Representative values from the current code

The thresholds make the measurement rules visible.

The logger does not just collect sensor output. It defines what counts as an acceptable mapping condition. Those choices show up directly in the constants and helper functions in the Arduino code.

ParameterCurrent valueRole in the survey
sonarCalSoundSpeed1500.0 m/sBaseline sound speed used when correcting sonar range.
sonarCalTempC20.0 °CReference temperature tied to that sound-speed baseline.
maxMapHdop1.8Rejects GPS fixes with poorer horizontal precision.
maxMapFixAgeMs2000 msRejects stale GPS fixes that no longer match the platform position closely.
maxMapSpeedKmph8.0 km/hLimits mapping when the platform is moving too quickly.
maxMapPitchDeg10.0°Rejects mapping points when platform tilt exceeds the allowed pitch range.
maxMapRollDeg10.0°Rejects mapping points when roll exceeds the allowed range.
