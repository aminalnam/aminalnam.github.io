Deployment Protocols

Deck Operations

Software engineers rarely have to worry about tying a bowline knot or avoiding seasickness while soldering. OMEGA bridging the physical gap requires rigorous deck protocols.

Lithium Fire Risk

Battery Safety

Water and lithium are a catastrophic combination. If a buoy hull fractures during deployment, saltwater ingress into an 18650 cell can trigger thermal runaway (a Class D metal fire). Nodes must never be stored inside a vessel's enclosed cabin during transit; they are tied on the open deck where they can be kicked overboard instantly if they begin smoking.

Mooring Rigging

Rigging & Knots

A 50kg steel anchor dropping to the seafloor generates immense shock force. Rodes must be rigged using proper marine knots (Bowline, Cleat Hitch) on stainless steel thimbles to prevent the nylon rope from chaffing entirely through in less than 48 hours.

Mooring Safe Working Load (SWL) 
The physics of rope degradation in seawater:

Nylon Breaking Strength (1/2"): ~3,000 kg
Safe Working Load (SWL) margin: 20%
Max Allowable Load: 600 kg

However, tying a knot physically weakens the fibers by forcing them 
to bend against each other under tension.
- Bowline Knot Efficiency: 60%
- Anchor Hitch Efficiency: 70%

If an Anchor Hitch is used:
New Breaking Strength = 3,000 kg * 0.70 = 2,100 kg
New SWL (20%) = 420 kg

When a 3-meter swell hits a buoy with high windage, the snap-load 
on the anchor can exceed 300 kg. If inappropriate knots (like a 
square knot, which has ~45% efficiency) are used, the rope will 
snap immediately. OMEGA mandates the Anchor Hitch, seized with 
stainless steel wire, to guarantee the 420 kg SWL.
Educational Deep Dive: Pre-Deployment

The Custom Firmware Builder.

A day before a major oceanic deployment, the last thing an engineer wants to do is manually hand-edit platformio.ini files, accidentally break a C++ pointer, and brick a sensor node. OMEGA eliminates this risk entirely through automated code generation.

The Core Concept

Automated Code Generation

The OMEGA Mission Portal features a built-in Custom Firmware Builder. Operators simply select their base hardware (e.g., ESP32-S3) and check off which sensors are physically wired to the board (e.g., BME280, u-blox GNSS). The Python Gateway dynamically templates the required C++ libraries, populates the setup() routines, and compiles the main.cpp firmware on the fly.

Implementation Guide

PlatformIO Integration

Once the operator approves the auto-generated code via the UI, the Gateway secretly shells out to the platformio CLI. It builds the binary payload in a secure temporary directory and flashes it directly to the tethered USB device—all without the operator writing a single line of C++.

Educational Deep Dive: Field Execution

The Desktop Mission Hub.

Cloud dashboards are useless 50 miles offshore without Starlink. For true operational independence, researchers rely on the Desktop Mission Hub—an offline-first, native desktop application.

The Core Concept

Offline-First Architectures

The Desktop Hub is built using the Tauri framework (Rust + React). Before the ship leaves the dock, the application preemptively downloads high-resolution Mapbox Vector Tiles and NOAA bathymetry charts for the deployment zone. It stores them in a local SQLite cache. When the ship loses internet, the React UI seamlessly swaps to the offline SQLite cache, ensuring the map never goes dark.

Implementation Guide

Native Hardware Access

Unlike a standard browser tab (which is sandboxed for security), the Tauri desktop wrapper has raw access to the host machine's filesystem and USB ports. This allows the offline portal to directly interface with tethered LoRa radios (via the Serial Port API) to pull live telemetry out of thin air, completely bypassing the cloud.

Educational Deep Dive: Remote Control

Mesh Command CLI & OTA Flashing.

Once a buoy is deployed, retrieving it to update its software is incredibly expensive. OMEGA solves this via a powerful Python CLI that beams administrative commands over 900MHz LoRa radio.

The Core Concept

RPC Over LoRa

The gateway.mesh_command CLI acts as a Remote Procedure Call (RPC) bridge. When an operator types a command to alter a buoy's transmission interval, the Python script encodes the instruction into a highly dense Binary TLV frame. It pushes the binary frame down a USB serial cable to a tethered "Relay Node", which broadcasts it into the LoRa mesh. The target buoy decodes the TLV frame and executes the command autonomously.

Implementation Guide

OTA Machine Learning

The most powerful CLI command is load-model. It reads a TensorFlow Lite Micro .tflite file from the host computer, shards the binary into 234-byte chunks, and sequentially beams them over LoRa. The edge node reassembles the chunks in flash memory, verifies the CRC32 checksum, and reboots—effectively learning a new Machine Learning capability while drifting in the ocean.

powershell
> python -m gateway.mesh_command --serial-port COM10 --target supreme-mesh-01 load-model ./models/sea_state.tflite
